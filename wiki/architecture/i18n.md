# i18n architecture

AsciiMark ships in **English (`en`)**, **Portuguese (`pt-BR`)**, and
**Spanish (`es`)**. All three apps (desktop, extension, site) read
from a single shared message catalog at `packages/i18n`.

## Stack

- **[Paraglide](https://inlang.com/m/gerre34r/library-inlang-paraglideJs)** — compile-time message generation. Each message becomes a typed JS function (`m.toolbar_open_folder()`); calls tree-shake per locale, so the production bundle only ships strings actually used.
- **Solid adapter** (`packages/i18n/src/solid.ts`) — lifts Paraglide's
  runtime locale into a Solid signal so JSX re-runs on switchLocale.
- **JSON message bundles** at `packages/i18n/messages/{en,pt-BR,es}.json` — single source of truth for all strings. Edited by hand (no CMS yet).

## The `(useLocale(), m.foo())` pattern

Paraglide messages are plain functions — they read the global locale
once when invoked. Without a Solid-side dep, JSX wouldn't re-render
on language switch. The convention every component uses:

```tsx
import * as m from "@asciimark/i18n";
import { useLocale } from "@asciimark/i18n/solid";

<button>{(useLocale(), m.toolbar_open_folder())}</button>
//          ^ tracks locale signal as a JSX dep
//                              ^ returns the translated string
```

The comma operator pattern is intentional:
- `useLocale()` registers the locale signal as a tracked dependency
  for the surrounding JSX node.
- `m.toolbar_open_folder()` returns the message string for the
  current locale.
- `switchLocale("pt-BR")` updates the signal → Solid re-renders any
  JSX node that read it → messages re-evaluate with the new locale.

**Domain rule:** every translated label MUST use the comma-operator
pattern. Calling `m.foo()` directly without `useLocale()` works on
first render but freezes — the label stays in the initial locale
forever. There's a vtest covering this in `empty-state.vtest.tsx`
(see "re-renders after a post-mount switchLocale").

## Locale detection

`packages/i18n/src/solid.ts::detectInitialLocale()` cascades:

1. `localStorage["asciimark-locale"]` — user choice persists across
   sessions.
2. `navigator.language` — exact match (`"pt-BR"`), then language-only
   (`"pt"` falls back to `"pt-BR"`).
3. `baseLocale` — `"en"`.

The Paraglide `strategy` is configured to `["globalVariable",
"preferredLanguage", "baseLocale"]` and the Solid adapter calls
`paraglideSetLocale(detected, { reload: false })` before the app
mounts. Reload is `false` because we drive re-render via the signal,
not a page refresh.

We DON'T use Paraglide's built-in `localStorage` strategy — it
hard-codes the key as `PARAGLIDE_LOCALE`, and we want our own
namespace (`asciimark-locale`).

## Adding a new locale

1. Create `packages/i18n/messages/<locale>.json` with the same keys as `en.json` (the parity check refuses to land an unbalanced bundle).
2. Update `packages/i18n/project.inlang/settings.json::locales`.
3. Run `bun --cwd packages/i18n run compile` to regenerate the runtime.
4. Add the locale to the Command Palette switcher in
   `apps/desktop/src/app.tsx` — the existing block iterates over
   `locales` from `@asciimark/i18n/solid`, so the only edit is the
   inline `loc === "..."` chain mapping locale code to a label key.
5. Add `command_language_<locale>` to all message bundles.

## Adding a new message

1. Add the key + value to `packages/i18n/messages/en.json` (source of truth).
2. Add a translation to **every** other locale file. Missing keys fail the parity check (`scripts/i18n-check.ts`).
3. Run `bun --cwd packages/i18n run compile` (or rely on `prepare` running on `bun install`).
4. Use the message in JSX with the comma-operator pattern.

## Parity check

`scripts/i18n-check.ts` runs in `pre-commit` and `pre-push`. It enforces:

- **Key parity** — every locale defines the same keys as `en`. Missing or extra keys fail.
- **Placeholder parity** — `{name}` placeholders in the base value are present (and not added) in translations. Catches "{n} files" → "arquivos" mistakes that render as literal `{n}`.

## What's intentionally NOT i18n'd

- **Site (`apps/site`)** — long-form prose in `guide-page.tsx` and
  `privacy-page.tsx`. Translating those would balloon the JSON
  bundles for low ROI. If we ever do, the right shape is route-based
  (`/pt-BR/guide`) with lazy-loaded page modules per locale.
- **`packages/core/src/keyboard-shortcuts.ts` descriptions** — those
  feed `ShortcutsHelp`. Migrating them would couple `core` to
  `i18n`, which we deliberately keep apart. The modal title is
  translated; the per-row descriptions stay in English (they're
  symbolic — `Cmd+P`, `Quick Open`).
- **Error messages and console logs** — devs read those, end users
  shouldn't see them.
- **Identifiers** — file paths, IDs, command IDs (`file.openFolder`),
  tab IDs, and routes. These are stable contracts.

## File-tree summary

```
packages/i18n/
├── package.json              # workspace package
├── project.inlang/
│   └── settings.json         # Paraglide config (locales, strategy)
├── messages/
│   ├── en.json               # source of truth (~90 keys)
│   ├── pt-BR.json
│   └── es.json
├── src/
│   ├── solid.ts              # adapter: useLocale, switchLocale, currentLocale
│   └── paraglide/            # GENERATED — gitignored, refreshed on `prepare` / `typecheck`

```

## Technical debt

Items deliberately deferred. Not blocking, not forgotten — pick up
when there's a real pull to address them.

### Site (`apps/site`) is NOT translated

Status: dependency wired (`@asciimark/i18n` is in `apps/site/package.json`),
zero strings migrated.

Why not: long-form docs in `guide-page.tsx` (~250 lines of prose) and
`privacy-page.tsx` (~120 lines, legal copy) would balloon the message
bundles for low ROI. Localized legal text also needs review — not
something to bulk-translate.

Three follow-up paths, in increasing scope:

1. **Minimum viable** (~30 min) — migrate Home labels only: nav, hero
   (kicker/title/description/CTAs), feature cards, downloads section
   labels, notice-panel. Add a `<select>` in `SiteLayout.tsx` calling
   `switchLocale()`. Guide + Privacy stay in English.
2. **Full content** (~3 h) — Guide + Privacy too. Adds ~300 keys.
   High maintenance overhead; needs a translator workflow before
   accepting.
3. **Route-based localization** (~6 h+) — `/pt-BR/guide`,
   `/es/privacy`. Rewrites TanStack Router setup, lazy-loads page
   modules per locale. Worth it only if SEO per-locale matters.

### `ShortcutsHelp` per-row descriptions stay English

`packages/core/src/keyboard-shortcuts.ts` defines descriptions
hardcoded in English (`"Open folder"`, `"New tab"`, etc.). Migrating
would couple `core` to `@asciimark/i18n`, which we deliberately keep
apart. The modal title is translated; the table cells are not.

Resolution path: change `description: string` to `descriptionKey:
string` in `ShortcutDescriptor`, then look up the key in the i18n
package from `ShortcutsHelp`. Keeps `core` decoupled.

### `tab-bar.tsx` context menu

`Close` / `Close Others` / `Close to the Right` / `Close All` are
hardcoded English. Low priority — context menus are short, the icons
help, and the user doesn't dwell here.

### MRU recent commands in Command Palette

Discussed in this session but not implemented. Today the palette
shows commands grouped (`File / View / Theme / …`) with alphabetical
sort within group — alphabetical changes by locale, which can confuse
users switching languages. A "Recently Used" tier on top would
stabilize the UX. Implementation sketch:

- `localStorage["asciimark-recent-commands"]` — array of command
  IDs, MRU first, capped at 5.
- `recordCommandUse(id)` called from `CommandPalette` `onSelect`.
- `filterCommands` prepends recent IDs when `query === ""`.

### TS declaration files for the i18n package

`paraglide-js compile` emits `.js` only by default. We can opt-in to
`.d.ts` via `--emit-ts-declarations`. Today consumers rely on
JSDoc-typed-JS imports working through `tsc --moduleResolution
bundler`, which works but isn't ideal. Switch the flag if any
downstream typecheck starts complaining.

## Lessons learned

Captured during the v0.8.0 i18n rollout. Each one cost real debugging
time — write them down so the next pass doesn't re-discover them.

### Paraglide v2 `compile` deletes the `--outdir`

The CLI clears the output directory before regenerating. The first
attempt put the adapter (`solid.ts`) in `src/` next to the generated
`runtime.js` — `bun run compile` then nuked it.

**Fix**: keep generated output in a subfolder (`src/paraglide/`) and
hand-written code in `src/`. The exports map in `package.json` points
at the right path. The generated subfolder gets its own `.gitignore`
that excludes everything (Paraglide creates it automatically).

### Paraglide's `localStorage` strategy hardcodes the key

Setting `strategy: ["localStorage", ...]` in `project.inlang/settings.json`
makes Paraglide read/write a key called `PARAGLIDE_LOCALE`. There's
no setting to rename it. If you want a custom key (e.g.
`asciimark-locale`), you need to manage `localStorage` yourself in
the adapter and use `strategy: ["globalVariable", ...]`.

### `setLocale(loc, { reload: false })` requires `globalVariable` strategy

By default the strategy is `["cookie", "globalVariable", "baseLocale"]`
(server-flavored). On a SPA without cookies, `setLocale` writes to
`document.cookie` and the next `getLocale()` doesn't see it. The fix
is to put `globalVariable` ahead — that strategy mutates an internal
`_locale` symbol that `getLocale()` reads back.

### Tauri dev caches the bundle harder than expected

After changing module-level state (locale runtime, render counters,
init constants), `window.location.reload()` re-runs the same
*cached* bundle. The new code only ships after restarting
`bun run dev:app`. Already documented in Round 5 of
[testing/strategies.md](../testing/strategies.md); reinforced here
because the same trap caught a "the i18n adapter doesn't work" two
hours of debugging.

### The `(useLocale(), m.foo())` pattern is non-obvious

Without the comma-operator, `m.foo()` evaluates once at render time
and freezes — JSX never re-renders on `switchLocale`. Pure-string
returns from a function look identical to a translated string at
first glance, so the failure is silent.

The vtest in `empty-state.vtest.tsx` ("re-renders after a post-mount
switchLocale") is the exact mutation-killer: it switches language
*after* mount and asserts the DOM updates. If a future agent drops
the comma-operator from a label, that test fails and `pre-commit`
blocks the commit.

### Translating long docs is the wrong cost

The instinct on "let's i18n the whole site" was wrong. Long docs are
high-churn (every release changes feature lists, screenshots, copy)
and translation must keep pace or rot silently. The right scope for
i18n bootstrap is *labels you can fit on a sticker* — buttons,
menus, palettes, status messages. Long-form prose needs a separate
workflow (translator, review, localized assets) that we don't have.

This is why the Site is in tech debt rather than half-translated.
Half-translated is worse than English-only — readers aren't sure
which version is canonical.

### Bundle-size matters for the extension

The Chrome MV3 extension has a 25 MB hard limit at the Web Store.
Today's bundle (~7.5 MB pre-gzip) has plenty of headroom, but
markdown-it / mermaid / asciidoctor add up. Paraglide's tree-shaking
of unused message keys is one of the reasons we chose it over a
runtime catalog: every screen ships only the strings it touches, and
the locale switch path doesn't pull every translation upfront.

The same logic justifies *not* migrating long docs to message
bundles — every dictionary entry adds bytes to every locale, even
the locales the current user will never load.
